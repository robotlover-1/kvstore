# kvstore 二开需求修改规划（基于 2026-04-21 需求清单）

## 1. 目标与范围

本次规划面向“第一次功能演示”所列出的 5 大能力：

1. 构建与工程规范（submodule、README、无警告编译）
2. 持久化能力（全量 + 增量）
3. 配置化启动（替代命令行）
4. 主从同步（历史数据 + 实时数据）
5. 大规模 key 超时（TTL）能力

同时遵循“一个功能一个测试用例、单用例数据量万级以上”的验收约束。

---

## 2. 需求拆解与模块映射

### 2.1 工程与文档

- **Submodule 管理**
  - 在主仓库中固定 ntyco 子模块版本，提供初始化脚本或 README 命令：
    - `git submodule update --init --recursive`
- **README 补全**
  - 增加：编译步骤、测试矩阵、性能测试方法与结果模板。
- **编译无告警**
  - 在 CI 与本地统一开启严格告警：`-Wall -Wextra -Werror`（按编译器兼容性调整）。

### 2.2 持久化：全量（Snapshot）

- **功能目标**
  - 存储 key-value 即可，不要求与增量日志命令格式一致。
- **技术路线**
  - 落盘：`io_uring` 异步写。
  - 加载：`mmap` 只读映射 + 顺序反序列化。
- **演示链路**
  - 插入 10w -> `save` -> 宕机 -> 重启加载 -> 校验 10w 数据一致。

### 2.3 持久化：增量（AOF/WAL）

- **功能目标**
  - 支持重启后重放增量日志，保证 10w 数据恢复。
- **技术路线**
  - 落盘：`io_uring` 批量提交（可按时间或条数 flush）。
  - 加载：`mmap` 读取日志 + 顺序回放。
- **演示链路**
  - 插入 10w -> 宕机 -> 重启回放 -> 校验 10w 数据一致。

### 2.4 配置文件化

- **功能目标**
  - 不依赖命令行参数；统一由配置文件控制。
- **配置项（最低集）**
  - `bind_ip`、`port`
  - `log_mode`
  - `persist_mode`（none/snapshot/aof/hybrid）
  - `role`（master/slave）
  - 主从地址（slave 模式必需）

### 2.5 主从同步

- **功能目标**
  - 历史数据同步 + 实时增量同步双通道。
- **技术路线（按需求预案）**
  - 历史数据：RDMA（bulk sync）
  - 实时数据：eBPF 辅助事件链路（增量复制）
- **演示链路**
  - master 先写 5w -> slave 启动并补齐 -> master 再写 5w -> slave 读到 10w 且一致。

### 2.6 大规模 TTL

- **功能目标**
  - 在大量过期 key 场景维持可接受延迟与吞吐。
- **建议方案**
  - 惰性删除 + 分片定时扫描（时间轮/小根堆择一）
  - 避免单次过期清理阻塞主线程，采用预算式批处理。

## 3. 交付阶段（建议 4 个里程碑）

## M1：工程基线与可复现环境

- 子模块固定与初始化脚本
- 配置文件解析框架
- 编译告警清零
- README 初版（构建 + 基本测试）

**验收标准**：
- 新环境按 README 在 30 分钟内完成构建和启动。
- `make`/`cmake` 全量构建无 warning。

## M2：持久化闭环（全量 + 增量）

- Snapshot 编解码 + `save` 指令
- 增量日志写入与回放
- `io_uring` 写盘通道
- `mmap` 加载通道

**验收标准**：
- 全量演示链路通过（10w）。
- 增量演示链路通过（10w）。

## M3：主从同步闭环

- slave 启动握手与角色状态机
- 历史数据同步（RDMA）
- 实时同步（eBPF 通道）

**验收标准**：
- 5w+5w 主从演示通过，slave 读取 10w 一致。

## M4：TTL 与性能收敛

- 大 key 数过期模型优化
- 压测脚本与性能报告
- README 完整版（含数据）

**验收标准**：
- 万级~十万级 TTL 压测无明显抖动尖峰。
- 报告可复现。

## 4. 测试规划（一个功能一个用例）

建议新增集成测试脚本（可放在 `tests/integration/`）：

1. `test_snapshot_100k.sh`
   - 插入 10w，save，重启恢复，逐条校验。
2. `test_aof_100k.sh`
   - 插入 10w，宕机恢复，回放校验。
3. `test_config_bootstrap.sh`
   - 纯配置文件启动，校验端口/日志/模式生效。
4. `test_repl_fullsync_incremental_100k.sh`
   - 5w 历史 + 5w 实时，slave 端全量一致性。
5. `test_ttl_massive_keys.sh`
   - 大量 TTL 写入 + 过期窗口观测（延迟/吞吐）。

每个脚本输出统一格式：
- case 名称
- 数据规模
- 耗时
- 成功条数/失败条数
- 最终 PASS/FAIL

## 5. 代码改造建议（按目录）

- `src/persistence/`
  - 拆分 `snapshot` 与 `aof` 子模块，隔离编解码协议。
- `src/main/kvstore.c`
  - 启动流程改为“先读配置再初始化模块”。
- `src/replication/`
  - 拆分全量同步与实时同步状态机，避免耦合。
- `src/expire/`
  - TTL 结构与扫描策略可配置。
- `tests/integration/`
  - 新增上述 5 个大规模用例与公共断言脚本。

## 6. 风险与缓解

1. **RDMA/eBPF 环境依赖高**
   - 缓解：提供 mock/fallback 模式，保证 CI 可跑核心逻辑。
2. **io_uring 在不同内核版本行为差异**
   - 缓解：探测内核能力；必要时回退 `pwrite`。
3. **10w 数据用例耗时过长**
   - 缓解：分层测试（quick/smoke/full），演示走 full。
4. **无告警编译与历史代码冲突**
   - 缓解：按模块分批清 warning，先主路径后边缘路径。

---

## 7. 建议优先级

- **P0（必须）**：配置文件、全量/增量持久化演示链路、测试用例、无告警编译。
- **P1（高）**：主从 10w 一致性演示闭环。
- **P2（优化）**：TTL 海量场景性能打磨、报告自动化产出。

以上规划可直接作为迭代任务拆分依据：每个里程碑拆成“开发任务 + 用例任务 + 文档任务”三类并行推进。
