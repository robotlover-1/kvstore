# Kernel 6.1 升级 + Tracepoint 捕获优化

## 背景

原 benchmark 在 kernel 5.15 上用 kprobe/kretprobe 挂钩 `tcp_recvmsg` 实现 eBPF 异步转发，QPS 始终落后于 sync 模式。设计文档提出用 tracepoint 替代 kprobe 降低每包开销，并升级内核到 6.1 以启用 fentry/fexit trampoline。

## 迭代过程

### 阶段 1: kernel 6.1 升级

**问题**: Ubuntu 20.04 官方源最高只到 5.15 HWE。

**解决**:
- 从 Ubuntu Mainline PPA 下载 v6.1.176 内核包（4 个 deb: headers-all, headers-generic, image, modules）
- kdprobe-headers-generic 依赖 libc >= 2.38（系统 2.31），安装失败但**不影响内核启动**，仅影响编译外部模块
- 安装后 `update-grub`，旧 5.15 内核保留在 GRUB 可回退
- RDMA rxe 模块确认包含在新内核中

### 阶段 2: 工具链适配

**问题 1**: bpftool 包装脚本只认识 `/usr/lib/linux-tools/<kernel-ver>/bpftool`，新版内核无对应目录。

**解决**: 从 git.kernel.org clone v6.1 源码，编译 `tools/bpf/bpftool`（v7.1.0），安装到 `/usr/lib/linux-tools/6.1.176-0601176-generic/`

**问题 2**: libbpf 0.5.0（系统自带）无法读取 kernel 6.1 BTF (`"failed to find valid kernel BTF"`)

**解决**: 从 kernel 6.1 源码编译 libbpf 1.1.0（`tools/lib/bpf`），修改 `Makefile.perf` 的 LDFLAGS/CFLAGS 指向新 libbpf

**问题 3**: vmlinux.h 需要重新生成适配 6.1

**解决**: `bpftool btf dump file /sys/kernel/btf/vmlinux format c` 生成 138K 行的新 vmlinux.h

### 阶段 3: fentry/fexit 尝试

**目标**: kernel 6.1 应支持 BPF trampoline，fentry/fexit 零陷阱开销。

**尝试 1**: 启用 fentry/fexit autoload
- **结果**: verifier 报 `reg type unsupported for arg#0 ... ptr_sock`
- **根因**: `BPF_PROG(fexit_recv, int retval, struct sock *sk, ...)` 中的 `ptr_sock` typed pointer 在 kernel 6.1 仍不被 verifier 支持
- kernel 6.2+ 才修复此限制

**尝试 2**: 去掉 `BPF_PROG`，改用原始 `u64 *ctx` 避免 BTF type tag
- **结果**: 程序加载成功，libbpf 显示 "using fentry/fexit trampoline"，但**完全不触发**（hit=0）
- **根因**: libbpf 1.1 的 SEC auto-attach 可能未正确注册 trampoline，或 `u64 *ctx` 签名破坏了自动链接

**结论**: fentry/fexit 不可用，回退到其他方案。

### 阶段 4: kprobe/kretprobe 尝试

**问题**: kprobe 程序加载成功（attach 无报错），但 `STAT_HIT=0`

**排查过程**:

1. 给 `kp_recv_entry` 开头加无条件 `stat_inc(STAT_HIT)` → hit 正常递增，**kprobe entry 确实触发了**
2. 给 `kp_recv_return` 的 msg 指针检查加 `STAT_MSG_NULL` → msg_null≈0，**per-CPU 传递正确**
3. 给 `read_iov_data` 加分步错误码 → rb_err/600 = step=1，即**第一个 `bpf_probe_read_kernel` 失败**

**根因**: `read_iov_data` 在 msg_ptr+40 处用 `bpf_probe_read_kernel` 读 iov 指针，kernel 6.1 上该调用返回错误。msghdr/iov_iter 结构布局虽然偏移相同（msg_iter@16, iov@24），但 kernel 6.1 的 `bpf_probe_read_kernel` 对此内存区域的访问有更严格校验。

**方案**: 需用 BPF CO-RE (`bpf_core_read`) 动态适配偏移，但 clang 10 不支持 CO-RE，需 clang >= 12。

**结论**: kprobe 模式在无 CO-RE 的 clang 10 下无法工作。

### 阶段 5: tracepoint 模式优化（重点）

tracepoint 挂钩 `syscalls:sys_{enter,exit}_read`，替代 kprobe 的 `tcp_recvmsg`。

#### 5.1 初始实现（v1）

**BPF 程序结构**:
```
sys_enter_read: ctl(ENABLED) + ctl(PID) + PID检查 + ctl(MASTER_FD) + fd检查 + HASH_MAP写入(buf+fd)
sys_exit_read:  ctl(ENABLED) + ret检查 + ctl(PID) + PID检查 + HASH_MAP读取(fd) + fd检查 + tmpbuf + probe_read_user + ringbuf
```

**问题**: 
- 5 个 map lookup / 每条 read
- HASH map (`enter_buf`) 有 spinlock 开销
- fd 过滤在 exit 侧，非匹配 read 仍走完完整路径

**结果**: QPS ~20K (latency 48µs)

#### 5.2 消除 HASH map（v2）

**改动**: 用 `entry_msg` per-CPU array (key=0:buf, key=1:fd) 替代 HASH map
- 2× per-CPU 写入（enter）+ 2× per-CPU 读取（exit），免锁

**结果**: QPS 仍 ~20K（无明显改善，瓶颈不在此）

#### 5.3 fd 过滤移到 enter + sentinel（v3）

**改动**:
- fd 过滤从 exit 移到 enter：不匹配的 fd 直接跳过，不保存 buf
- CTL_MASTER_FD 初始化为 ~0UL sentinel → enter 过滤所有 read，accept 后更新为真实 fd

**结果**: 
- fwd=5,500（确切匹配请求数），反馈循环消除 ✓
- QPS 仍 ~20K

**瓶颈分析**: 每条非匹配 read 仍走 3 个 ctl lookup（ENABLED + PID + MASTER_FD）；匹配 read 走全部 5 个 lookup。

#### 5.4 合并 lookup（v4，最终版）

**改动**:
1. **CTL_PID 编码 enabled+PID**: `if (!pid_val) return 0` → 1 次 lookup 搞定开关+过滤
2. **fd+PID 编码同一 u64**: `(master_fd << 32) | pid` → fd 过滤 0 次额外 lookup
3. **exit 不查 PID**: enter 已过滤，exit 只读 buf → exit 从 2 lookup 减到 0
4. **哨兵 ~0U**: loader 初始化 `(~0U << 32) | pid`，确保 accept 前过滤所有 fd

**最终 BPF 程序**（每条 read 的开销）:
```
非匹配 PID:   1 lookup (CTL_PID=0) → return
匹配 PID, fd不匹配: 1 lookup + pid提取 + fd提取 + fd比较 → STAT_SKIP_FD
匹配 PID+fd:  1 lookup + pid提取 + fd提取 + fd比较 + 1 per-CPU写入(buf)
exit (匹配PID+fd):  per-CPU读buf + tmpbuf + probe_read_user + ringbuf reserve+copy+submit
```

5 个 lookup → **1 个 lookup**。

**结果**:
- fwd=5,500（反馈循环消除）✓
- QPS ~19K (latency 52µs)

### 额外修复

- **SIGPIPE**: `write_full` 写入已关闭的 slave socket 触发 SIGPIPE 杀死进程。修复：`main()` 中 `signal(SIGPIPE, SIG_IGN)` + `write_full` 处理 EPIPE

## 最终 Benchmark

kernel 6.1.176, count=5000:

| payload | none   | sync   | tp      | tp fwd |
|---------|--------|--------|---------|--------|
| 64B     | 28,004 | 36,825 | 19,116  | 10,986 |
| 512B    | 26,557 | 31,537 | 19,006  | 5,500  |
| 4096B   | 26,281 | 31,905 | 19,348  | 5,500  |

tp QPS ≈ 72% of none 基线，一致稳定在 ~19K。剩余开销来自：
- `bpf_probe_read_user`（每匹配 read 一次用户态拷贝）
- ringbuf reserve+copy+submit
- 每个 `read()` syscall 触发 2 次 BPF 执行（enter+exit）

## 未解决问题

| 项目 | 原因 | 修复路径 |
|------|------|----------|
| kprobe 模式 | `read_iov_data` 硬编码偏移在 kernel 6.1 上 `bpf_probe_read` 失败 | 装 clang >= 12 启用 CO-RE 或更新偏移 |
| fentry/fexit | verifier 拒 `ptr_sock` typed pointer | 需 kernel 6.2+ 或 clang CO-RE |
| tp QPS < none | 每 read() 触发 enter+exit BPF | 可用 raw tracepoint 或 filter at kernel level |
