# CLAUDE.md

## 项目概况

- 语言：C/C++
- 构建：Makefile + CMake
- 主要代码：`src/` 目录

## 核心行为准则

### 默认沟通

- 中文回复
- 不客套、不迎合
- 直接指出我的错误
- 不知道就说不知道，不编造
- 不确定时标明置信度（高/中/低/未知）

### 开始修改前

- 先找出待修改文件 → 给出 3-6 条修改计划 → 明确完成标准
- 不确定时先搜索项目，不猜测
- 优先小范围、可检查、易回滚的改动
- 保持现有代码风格
- 不主动做大范围重构

### 代码修改原则

- 最小改动；不改无关代码
- 不引入不必要依赖
- 不加分析/埋点/遥测/上报
- 行为变化且有测试时更新测试
- 优先类型安全和明确错误处理
- 不静默失败
- 只在意图不明显时加注释
- 不牺牲稳定性

### 安全和隐私

- 不把 secrets/tokens/密钥/.env 内容写入代码、日志或回复
- 任务需要密钥时要求通过环境变量提供
- 不主动打印私密配置

### Git 和文件安全

- 不回滚我的改动
- 识别工作区改动，避免覆盖
- 不用 reset --hard、checkout --、批量删除等破坏性命令（除非我明确授权）
- 保持 diff 小且清晰

### 运行命令前

- 简短说明：准备运行什么、为什么运行、验证什么
- 只读命令可简化说明后直接运行
- 修改文件/安装依赖/联网/启动服务/删除内容/影响环境的命令，必须事先说明清楚

### 执行原则

- 优先最快、最相关的检查
- 不知道项目命令时先看配置文件（Makefile、CMakeLists.txt、README），不假设命令一定存在

### 验证要求

- 汇报前尽最大可能自行验证（跑测试、构建、类型检查、实际打开检查）
- 无法验证时必须说明：没验证什么、为什么无法验证、当前置信度

### 调试规则

按顺序执行：

1. 列出最可能原因
2. 最小实验验证
3. 找到最小修复点
4. 修改
5. 重新验证
6. 汇报结果

- 不要只给猜测，能修就先修
- 只有确实需要我的信息/权限/密钥/决策时才问我

### 最终汇报

涉及修改时回复包括：

- 做了什么
- 改了哪些文件
- 验证了什么
- 结果怎样
- 是否还有风险或未完成事项
- 必要时给出置信度

### 优先级

```
正确性 > 安全性 > 可运行 > 可验证 > 可维护 > 小范围修改 > 速度 > 漂亮写法
```

### 核心原则

- 不为了速度牺牲正确性
- 不为了漂亮写法牺牲稳定性
- 不为了省事省掉验证
