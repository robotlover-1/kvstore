下面我不再按你列的 6 个点机械拆分，而是按“**如果我要设计并实现这个 KV 系统，我会怎么一步步搭起来**”的思路来讲。这样你会更容易理解：

1. 先解决什么问题
2. 为什么这么设计
3. 代码里具体怎么落地
4. 后面的功能又是怎么复用前面的能力的

我会把这 6 个功能自然串起来。

---

# 1. 先定系统目标：这不是单纯的字典，而是一个“可联网、可持久化、可复制”的 KV 服务

这个项目的真正目标，不只是提供 `set/get`，而是实现一个小型 Redis 风格服务。
从 `main()` 的初始化顺序就能看出来，作者不是先写某个数据结构，而是先搭出一个完整服务的骨架：

```c
kvs_array_create(&global_array);
kvs_rbtree_create(&global_rbtree);
kvs_hash_create(&global_hash);
kvs_expire_create(&global_expire);
if (persist_init() != 0) { ... }
persist_recover();
return reactor_start();
```



这段流程非常关键，它说明系统启动时依次完成了几件事：

* 初始化三套 KV 引擎
* 初始化过期表
* 初始化持久化文件
* 从 dump + aof 恢复数据
* 启动网络事件循环，正式对外服务

所以这个项目的设计中心是：

**网络请求 → 协议解析 → 命令执行 → 存储引擎 → TTL/持久化/复制联动**

后面所有功能，都是围绕这条主链增加能力。

---

# 2. 第一层：先把“命令处理中心”搭起来，而不是先纠结底层数据结构

如果你自己写这种系统，第一步其实不是红黑树、哈希表，而是先定义一个“统一命令入口”。
这个项目里，这个角色就是：

* `parse_resp_stream()`：负责解析 RESP 协议流
* `handle_parsed_command()`：负责执行一条已经解析好的命令

都在 `kvstore.c` 里。

## 2.1 RESP 解析器支持批量处理

它读取的是一整段 TCP 输入缓冲，而不是只读一条命令：

```c
int parse_resp_stream(conn_t *c, unsigned char *buf, size_t *len, int from_replication)
{
    size_t pos = 0;
    while (pos < *len) {
        ...
        handle_parsed_command(c, argc, argv, argl, buf + start, p - start, from_replication);
        ...
        pos = p;
    }
    ...
}
```



这段代码的设计意义很大：

* 一个 socket 一次 `recv()` 可能收到半条命令，也可能收到多条命令
* 所以必须支持“流式解析”
* `while (pos < *len)` 说明它天然支持 **pipeline / 批量命令**

这就是你列出来的“批量处理”能力的核心来源。
它不是额外单独写了一套“批处理模块”，而是**协议解析器本身就支持一次处理多条命令**。

## 2.2 为什么命令处理要集中在 `handle_parsed_command()`

因为系统后续的多个高级能力都要挂在这里：

* 普通 KV 操作
* TTL 联动
* AOF 追加
* 主从复制广播
* 从库只读限制
* 保存快照
* 查询内存状态

看这段代码就明白了：

```c
if (rc == 0) {
    n = resp_simple_string(resp, sizeof(resp), "OK");
    if (!from_replication && is_write_cmd(cmd)) {
        persist_append_raw(raw, rawlen);
        if (g_cfg.role == ROLE_MASTER) repl_broadcast(raw, rawlen);
    }
}
```



这实际上是整个系统的“事务总线”：

* 命令先执行成功
* 成功后再进入持久化
* 然后再复制给从节点

也就是说，**持久化和复制不是各自独立去监听变化，而是统一由命令执行层触发**。
这种设计非常适合教学和小型系统，逻辑很集中，容易维护。

---

# 3. 第二层：复杂 KV 存储不是“一个存储结构”，而是“统一接口 + 三种引擎”

你说的“复杂 KV 存储”，真正的复杂点不在某一种结构，而在于：

**系统同时支持三种不同引擎，并且对上层命令透明。**

## 3.1 引擎选择机制

`kvstore.c` 里先通过命令前缀决定使用哪个引擎：

```c
static int cmd_engine(const char *cmd) {
    if (cmd[0] == 'R') return KVS_ENGINE_RBTREE;
    if (cmd[0] == 'H') return KVS_ENGINE_HASH;
    return KVS_ENGINE_ARRAY;
}
```



这意味着：

* `SET/GET/DEL` 默认走 array
* `RSET/RGET/RDEL` 走红黑树
* `HSET/HGET/HDEL` 走哈希表

然后再把命令前缀去掉：

```c
static const char *strip_prefix(const char *cmd) {
    if (cmd[0] == 'R' || cmd[0] == 'H') return cmd + 1;
    return cmd;
}
```



这样就把“引擎选择”和“具体操作类型”拆开了。

## 3.2 统一引擎抽象接口

项目没有搞复杂的函数指针表，而是直接写了统一分发函数：

```c
static int engine_set(int engine, char *key, char *value) { ... }
static int engine_mod(int engine, char *key, char *value) { ... }
static int engine_del(int engine, char *key) { ... }
static char *engine_get(int engine, char *key) { ... }
static int engine_exist(int engine, char *key) { ... }
```



这个设计的好处是：

* 上层命令逻辑只处理 `engine + op`
* 下层每种结构自己负责内部细节
* TTL、复制、持久化都不需要知道底层到底是数组、树还是哈希

所以“复杂 KV 存储”的本质不是三种结构都实现了，而是**三种结构被统一成了一套标准接口**。

---

# 4. 第三层：三种存储引擎分别怎么实现

---

## 4.1 Array：最简单的基线实现

`kvs_array.c` 的核心结构就是一个固定大小数组：

```c
inst->table = (kvs_array_item_t *)kvs_malloc(KVS_ARRAY_SIZE * sizeof(kvs_array_item_t));
```



每个槽位存：

* `key`
* `value`

查找是线性扫描：

```c
static int find_slot(kvs_array_t *inst, char *key) {
    for (int i = 0; i < KVS_ARRAY_SIZE; ++i) {
        if (inst->table[i].key && strcmp(inst->table[i].key, key) == 0) return i;
    }
    return -1;
}
```



### 设计意图

这套实现非常朴素，但有明确意义：

* 作为最容易理解、最直接的 KV 实现
* 方便验证协议、网络、TTL、持久化是否工作
* 给后面更复杂结构提供一个对照基准

### 技术细节

`set` 时会扫描第一个空槽位：

```c
if (!inst->table[i].key) {
    inst->table[i].key = kvs_malloc(klen + 1);
    inst->table[i].value = kvs_malloc(vlen + 1);
    ...
}
```



`del` 时释放 key/value，并把槽位清空。
这个设计简单，但也意味着：

* 查找 O(n)
* 删除后产生空洞
* 没有压缩整理

所以它更像“演示型引擎”而不是高性能方案。

---

## 4.2 Hash：实际可用的高效 KV 结构

`kvs_hash.c` 使用的是 **链地址法哈希表**。

### 哈希函数

不是简单字符求和，而是 FNV 风格混合：

```c
unsigned int sum = 2166136261u;
for (int i = 0; key[i] != 0; ++i) {
    sum ^= (unsigned char)key[i];
    sum *= 16777619u;
}
return (int)(sum % (unsigned int)size);
```



这比最原始的加和 hash 更均匀。

### 冲突解决

桶数组里挂链表：

```c
new_node->next = hash->nodes[idx];
hash->nodes[idx] = new_node;
```



即头插法。

### 技术细节

插入前先查重：

```c
for (hashnode_t *node = hash->nodes[idx]; node; node = node->next) {
    if (strcmp(node->key, key) == 0) return 1;
}
```



修改时重新分配 value 内存，而不是原地覆盖：

```c
char *nv = (char *)kvs_malloc(vlen + 1);
...
kvs_free(node->value);
node->value = nv;
```



这样做更安全，避免长度变化导致越界。

### 设计意义

这是系统里最适合一般 KV 场景的结构：

* 平均查找 O(1)
* 删除方便
* 与 TTL、持久化、复制集成成本低

---

## 4.3 RBTree：有序 KV 存储

`kvs_rbtree.c` 是最复杂的底层实现。

它包含标准红黑树的所有基本操作：

* 左旋 `rbtree_left_rotate`
* 右旋 `rbtree_right_rotate`
* 插入修复 `rbtree_insert_fixup`
* 删除修复 `rbtree_delete_fixup`
* 查找 `rbtree_search`

### 为什么系统需要红黑树

因为数组和哈希表都不保序，而红黑树可以：

* 维持 key 的有序性
* 支持中序遍历导出
* 给将来的范围查询、排序类命令留接口

### 关键技术点：nil 哨兵节点

创建时：

```c
inst->nil = (rbtree_node*)kvs_malloc(sizeof(rbtree_node));
inst->nil->color = BLACK;
inst->root = inst->nil;
```



这是一种经典实现方式，能减少空指针判断。

### 插入逻辑

先按 BST 规则定位，再统一做红黑修复：

```c
z->left = T->nil;
z->right = T->nil;
z->color = RED;
rbtree_insert_fixup(T, z);
```



### 删除逻辑

先删除，再根据颜色决定是否修复：

```c
if (y->color == BLACK) {
    rbtree_delete_fixup(T, x);
}
```



### 封装成 KV 接口

文件后半部分又把红黑树包装成：

* `kvs_rbtree_set`
* `kvs_rbtree_get`
* `kvs_rbtree_mod`
* `kvs_rbtree_del`
* `kvs_rbtree_exist`

这样才和 array/hash 接口统一。

# 5. 第四层：有了 KV 引擎之后，怎么给 key 加上“定时删除”能力

如果只是普通 KV，做到这里就差不多了。
但一旦要支持 TTL，就必须回答一个问题：

**过期时间存在哪里？**

这个项目的设计不是把 expire 时间硬塞进 array/hash/rbtree 的每个节点里，而是专门维护了一张独立的过期表。
这是一个很好的设计。

---

## 5.1 过期信息独立存储

`kvs_expire.c` 定义了全局过期表：

```c
kvs_expire_table_t global_expire = {0};
```



内部是一个 hash bucket 结构，键不是单纯的 `key`，而是 `(engine, key)`：

```c
static unsigned int expire_hash(const char *key, int engine) {
    ...
    h ^= (unsigned int)engine;
    ...
}
```



这点非常重要，因为：

* array 里可能有 key = "a"
* hash 里也可能有 key = "a"
* rbtree 里也可能有 key = "a"

如果 TTL 表只按 key 建索引，会冲突。
所以这里把 `engine` 混进 hash，是为了区分三套命名空间。

---

## 5.2 设置过期时间

```c
long long expire_at = kvs_now_ms() + ttl_ms;
```



调用点在 `handle_parsed_command()` 里：

```c
else if (!strcmp(op, "EXPIRE") && argc == 3) {
    ...
    rc = kvs_expire_set(&global_expire, engine, argv[1], atoll(argv[2]) * 1000);
}
```



也就是说，客户端命令给的是秒，内部统一转换成毫秒。

---

## 5.3 TTL 查询

```c
long long ttl = kvs_expire_ttl(&global_expire, engine, argv[1]);
```



内部返回语义：

* `-1`：没有设置过期
* `-2`：已经过期
* 正数：剩余秒数

而且这里还做了向上取整：

```c
long long s = remain_ms / 1000;
if (remain_ms % 1000) s += 1;
```



这和 Redis 的 TTL 显示风格很像，更符合用户直觉。

---

## 5.4 为什么 TTL 需要“被动删除 + 主动删除”两套机制

单靠一种机制都不够。

### 被动删除

每次访问 key 之前，先检查是否过期：

```c
static int try_expire(int engine, char *key) {
    if (kvs_expire_is_expired(&global_expire, engine, key)) {
        engine_del(engine, key);
        kvs_expire_del(&global_expire, engine, key);
        return 1;
    }
    return 0;
}
```



调用点很多，例如：

```c
try_expire(engine, argv[1]);
char *v = engine_get(engine, argv[1]);
```



这叫 lazy expire，优点是简单，访问时才清理。
但问题是，如果 key 过期了却一直没人访问，它会一直占内存。

### 主动删除

所以又有定期扫描：

```c
int kvs_active_expire_cycle(int budget)
```



在 epoll 主循环里每隔约 100ms 扫一次：

```c
if (now - g_last_expire >= 100) {
    kvs_active_expire_cycle(32);
    g_last_expire = now;
}
```



这里 `budget=32` 表示每轮最多删 32 个，避免一次扫描阻塞事件循环太久。

### 这两套机制如何配合

* 被动删除保证“访问到的 key 绝不会返回过期数据”
* 主动删除保证“冷数据过期后也会被最终清掉”

这就是你列出的“key 的定时删除”的完整实现思路。

# 6. 第五层：既然是服务，就必须考虑内存管理，而且这个项目做成了可切换策略

这个项目一个比较亮眼的地方，是没有直接全程裸用 `malloc/free`，而是抽象出一套统一内存接口：

* `kvs_malloc`
* `kvs_calloc`
* `kvs_realloc`
* `kvs_free` 

这样上层所有模块，包括：

* array
* hash
* rbtree
* expire
* reactor 输出队列

都统一通过这套接口申请内存。
这就为“切换内存策略”打下了基础。

---

## 6.1 三种内存策略怎么定义

`kvs_mem.c` 里定义了三种后端：

```c
#define KVS_MEM_LIBC 0
#define KVS_MEM_JEMALLOC 1
#define KVS_MEM_CUSTOM 2
```



然后通过名字映射：

```c
if (!strcmp(name, "libc")) return KVS_MEM_LIBC;
if (!strcmp(name, "jemalloc")) return KVS_MEM_JEMALLOC;
if (!strcmp(name, "custom")) return KVS_MEM_CUSTOM;
```



启动参数由 `parse_args()` 读入：

```c
else if (!strcmp(argv[i], "--mem") && i + 1 < argc)
    snprintf(g_cfg.mem_backend, sizeof(g_cfg.mem_backend), "%s", argv[++i]);
```



最后 `main()` 中初始化：

```c
if (kvs_mem_init(g_cfg.mem_backend) != 0) { ... }
```



所以切换方式是：

```bash
./kvstore --mem libc
./kvstore --mem jemalloc
./kvstore --mem custom
```

---

## 6.2 libc 模式

最直接，就是系统默认分配器。
在 `backend_malloc()` 里：

```c
case KVS_MEM_LIBC:
default:
    return malloc(size);
```



这是默认后端，也是最稳定的基线。

---

## 6.3 jemalloc 模式

这个模式不是简单调用 jemalloc API，而是通过 `LD_PRELOAD` 重新拉起进程，让整个进程的 `malloc/free` 都走 jemalloc：

```c
static int ensure_jemalloc_process_image(const char *argv0, char **argv) {
    ...
    setenv("LD_PRELOAD", preload, 1);
    setenv(KVS_JEMALLOC_ACTIVE_ENV, "1", 1);
    execvp(argv0, argv);
}
```



这是一个很实用的技巧：

* 不需要在代码里手工改成 `je_malloc`
* 直接把进程级分配器切过去
* 对项目现有代码侵入很小

`main()` 中如果检测到 `--mem jemalloc`，会先调用 `kvs_mem_prepare_process()` 处理这件事。

---

## 6.4 custom 模式：这个项目最有技术含量的内存实现

自定义内存池采用的是“小对象 slab + 大对象 mmap + 兜底 malloc”的组合。

### 小对象：分级 slab

有 8 个大小等级：

```c
{32}, {64}, {128}, {256}, {384}, {512}, {768}, {1024}
```



对于 `<= 1024` 字节的分配，先找到最小可容纳的 class：

```c
static int class_index_for(size_t sz) {
    for (int i = 0; i < SMALL_CLASS_COUNT; ++i) {
        if (sz <= g_mem.classes[i].size) return i;
    }
    return -1;
}
```



如果 free list 不够，就扩一页 slab：

```c
void *mem = mmap(NULL, alloc_size, PROT_READ | PROT_WRITE, MAP_PRIVATE | MAP_ANONYMOUS, -1, 0);
```



然后把这一页切成多个 chunk，挂入 free_list。

分配时直接从 free_list 取：

```c
small_chunk_t *chunk = g_mem.classes[idx].free_list;
g_mem.classes[idx].free_list = chunk->next;
```



释放时再挂回去：

```c
chunk->next = g_mem.classes[idx].free_list;
g_mem.classes[idx].free_list = chunk;
```



### 大对象：mmap

超过 `SMALL_MAX_SIZE` 的对象，不走 slab，而是直接页对齐 mmap：

```c
size_t rounded = (total + pagesz - 1) / pagesz * pagesz;
large_hdr_t *hdr = (large_hdr_t *)mmap(NULL, rounded, ...);
```



这样做的原因是：

* 大对象不适合塞进 slab
* 直接独立映射，释放时 `munmap` 更干净

### fallback

如果 custom 路径失败，比如 slab 扩展失败，就退回普通 malloc：

```c
return fallback_malloc(size);
```



这保证了自定义分配器不会因为局部问题把整个服务搞挂。

---

## 6.5 为什么这个内存模块不只是“分配”，还做了统计

`kvs_mem_get_stats()` 会导出很多指标：

* 当前已申请次数
* small alloc/free 次数
* 当前 small 使用量
* large mmap 总量
* 内部碎片率
* page 利用率
* 每个 class 的 chunk/page 情况 

然后通过命令 `MEMSTAT` 暴露给客户端：

```c
if (!strcmp(cmd, "MEMSTAT")) {
    char info[BUFFER_CAP];
    n = build_memstat_text(info, sizeof(info));
    ...
}
```



也就是说，这个内存模块不只是“能切换”，还支持观察和评估不同策略效果。

# 7. 第六层：为什么要先实现持久化，再实现主从复制

因为主从复制里有“全量同步”，而全量同步本质上就依赖“把当前内存状态导出成一份完整数据”。

这个项目里，全量导出的能力其实不是先为复制写的，而是先为持久化写的。
这一点很关键。

---

## 7.1 增量持久化：AOF

`kvs_persist.c` 里打开 AOF 文件：

```c
g_aof_fp = fopen(g_cfg.aof_path, "ab+");
```



每次写命令成功后：

```c
persist_append_raw(raw, rawlen);
```



它直接把原始 RESP 命令追加进去：

```c
if (fwrite(buf, 1, len, g_aof_fp) != len) return -1;
fflush(g_aof_fp);
```



### 为什么直接写 RESP 原始命令很妙

因为：

* 网络协议本来就是 RESP
* 内部恢复时也有 RESP 解析器
* 复制传播时广播的也是 RESP 原始命令

于是同一种命令格式，被同时复用于：

* 客户端请求
* AOF 持久化
* 主从增量复制

这让整个系统非常统一。

---

## 7.2 全量持久化：dump 快照

`persist_save_dump()` 调用：

```c
int rc = kvs_snapshot_to_fp(fp);
```



而 `kvs_snapshot_to_fp()` 在 `kvstore.c` 中实现。

它分别遍历三种引擎：

* `snapshot_array(fp)`
* `snapshot_rbtree_node(fp, ...)`
* `snapshot_hash(fp)`

每个 key/value 都被重新编码成 RESP 命令写入文件，例如：

```c
emit_cmd3_fp(fp, "SET", key, value)
emit_cmd3_fp(fp, "RSET", key, value)
emit_cmd3_fp(fp, "HSET", key, value)
```



### TTL 怎么进入快照

作者没有把绝对过期时间直接写进文件，而是写成剩余 TTL：

```c
long long ttl = kvs_expire_ttl(&global_expire, engine, key);
...
const char *cmd = engine == KVS_ENGINE_ARRAY ? "EXPIRE" : ...
return emit_cmd3_fp(fp, cmd, key, sec);
```



也就是说，快照中如果某个 key 还有过期时间，就会额外追加一条 `EXPIRE/REXPIRE/HEXPIRE` 命令。

这个实现很简洁，恢复时无需特殊分支，只要重新走命令执行逻辑即可。

---

## 7.3 恢复流程：先 dump，再 aof

启动恢复：

```c
int persist_recover(void) {
    replay_file(g_cfg.dump_path);
    replay_file(g_cfg.aof_path);
    return 0;
}
```



顺序是：

1. 先加载全量 dump
2. 再回放后续 AOF 增量命令

### 为什么顺序必须这样

因为 dump 表示某一时刻的完整状态，AOF 表示此后的变更。
如果顺序反了，后加载 dump 会把 AOF 的更新覆盖掉。

### replay 的实现

它读文件内容后直接塞回 RESP 解析器：

```c
parse_resp_stream(NULL, buf, &len, 1);
```



`from_replication = 1` 的这个思路很有意思：
恢复过程、本质上和“从主库收到命令流”很像，所以作者复用了同一套处理链。

---

# 8. 第七层：主从复制其实是“持久化能力”的自然延伸

当你已经有：

* 完整快照导出能力
* RESP 原始命令流
* 命令统一执行入口

那么主从复制就很好做了。

---

## 8.1 从节点如何同步主节点

`kvs_repl.c` 中，从节点会起一个后台线程：

```c
int start_slave_thread(void) {
    pthread_t tid;
    pthread_create(&tid, NULL, slave_thread, NULL);
    pthread_detach(tid);
}
```



线程里死循环：

1. 连接主节点
2. 发送 `REPLSYNC`
3. 持续接收主节点发来的数据
4. 把收到的数据送进 `parse_resp_stream(NULL, buf, &blen, 1)` 执行 

这说明从节点并没有单独维护一个“复制协议解析器”，而是**直接把主节点发来的东西当成 RESP 命令流执行**。

---

## 8.2 主节点如何识别复制连接

当主节点在命令处理中心收到：

```c
if (!strcmp(cmd, "REPLSYNC")) { 
    repl_add_slave(c); 
    queue_snapshot(c); 
    return 0; 
}
```



就把当前连接注册进副本链表：

```c
c->is_replica = 1;
c->next_replica = g_replicas;
g_replicas = c;
```



所以复制本质上就是：
**某些普通连接被标记成 replica，后面写命令会额外广播给这些连接。**

---

## 8.3 全量同步怎么做

`queue_snapshot(c)` 的流程是：

1. 先发一个 `+FULLRESYNC`
2. 生成 dump 文件
3. 读取 dump 文件内容并排队发送给 slave
4. 最后发 `REPLDONE` 

代码：

```c
int n = resp_simple_string(hdr, sizeof(hdr), "FULLRESYNC");
queue_bytes(c, ...);

FILE *fp = fopen(g_cfg.dump_path, "wb");
kvs_snapshot_to_fp(fp);
...
while ((r = fread(buf, 1, sizeof(buf), fp)) > 0) queue_bytes(c, buf, r);

size_t done = resp_build_cmd1(buf, sizeof(buf), "REPLDONE");
queue_bytes(c, buf, done);
```



### 这里为什么用了 `queue_bytes()`

因为网络层是非阻塞的，不能直接 `send()` 一把写死。
所以所有复制数据也统一走连接输出队列，这和普通命令响应共用一套发送机制。

---

## 8.4 增量复制怎么做

这就是前面说的统一命令总线：

```c
if (!from_replication && is_write_cmd(cmd)) {
    persist_append_raw(raw, rawlen);
    if (g_cfg.role == ROLE_MASTER) repl_broadcast(raw, rawlen);
}
```



主节点把原始写命令原封不动广播：

```c
for (conn_t *c = g_replicas; c; c = c->next_replica)
    queue_bytes(c, raw, rawlen);
```



所以增量复制和 AOF 一样，都是基于原始 RESP 命令流。

这套设计很漂亮：

* AOF 追加的是写命令
* replication 广播的也是写命令
* recover 和 slave apply 都靠 `parse_resp_stream`

**一个协议流，复用了三次。**

---

## 8.5 从库只读限制

从节点不能接受写请求，这在 `handle_parsed_command()` 里做了限制：

```c
if (g_cfg.role == ROLE_SLAVE && !from_replication && is_readonly_slave_blocked(cmd)) {
    n = resp_error(resp, sizeof(resp), "read only slave");
    ...
}
```



也就是：

* 来自主库同步流的写命令可以执行
* 客户端直接发给从库的写命令会被拒绝

这保证了主从一致性。

---

# 9. 第八层：网络层为什么选 reactor，以及它如何支撑前面所有功能

到这里你会发现，前面所有能力最后都需要网络层来托底：

* 客户端请求要收发
* 主从复制数据要收发
* 全量同步文件内容要分块发
* 普通响应也要异步发

所以网络层必须支持：

* 非阻塞 IO
* 输入输出缓冲
* 事件驱动
* 与 TTL 定时任务共存

这个项目选择的是 `epoll + reactor`，在 `reactor.c` 里实现。

---

## 9.1 连接对象设计

每个 fd 对应一个 `conn_t`，并由 `fdmap[65536]` 映射：

```c
static conn_t *fdmap[65536];
```



这样事件到来时，可以 O(1) 找到连接上下文。

连接里至少维护两块关键状态：

* 输入缓冲 `inbuf + in_len`
* 输出队列 `out_head/out_tail`

这正是 RESP 流式解析和非阻塞发送所需要的。

---

## 9.2 为什么需要输出队列 `queue_bytes()`

非阻塞 socket 不保证一次写完，所以项目没直接在命令执行后 `send()`，而是把数据挂入链表：

```c
out_node_t *n = (out_node_t *)kvs_malloc(sizeof(*n));
n->data = (unsigned char *)kvs_malloc(len);
memcpy(n->data, buf, len);
...
if (c->out_tail) c->out_tail->next = n;
else c->out_head = n;
c->out_tail = n;
```



然后打开 `EPOLLOUT`：

```c
mod_events(c, EPOLLIN | EPOLLOUT);
```



这套机制让：

* 普通响应
* 全量复制快照数据
* 增量复制命令流

都能统一异步发送。

---

## 9.3 读流程

`on_read()` 中不断 `recv()`，读到的数据累积到输入缓冲里，然后调用：

```c
parse_resp_stream(c, c->inbuf, &c->in_len, 0);
```

这正是前面说的协议解析总入口。

读完后，如果解析过程中生成了响应，则打开写事件。

---

## 9.4 写流程

`on_write()` 负责把输出链表里的节点逐个发出去：

```c
ssize_t w = send(c->fd, n->data + n->sent, n->len - n->sent, 0);
```



如果只发了一部分，就记录 `sent`，下次继续。
如果一个 node 全发完了，就释放。

这就是非阻塞网络程序的标准写法。

---

## 9.5 主事件循环如何把“定时删除”和“网络 IO”结合起来

最外层：

```c
int n = epoll_wait(g_epfd, events, MAX_EVENTS, 100);

long long now = kvs_now_ms();
if (now - g_last_expire >= 100) {
    kvs_active_expire_cycle(32);
    g_last_expire = now;
}
```



也就是说，事件循环不是只做网络事件，还把 TTL 的后台清理一起调度了。
这样避免单独再开一个专门的 expire 线程，整体更简单。

---

# 10. 把 6 个功能串起来看：它们不是平铺的，而是层层依赖的

你要求我讲“各个功能如何实现，以及功能之间的依赖”，这里给你按依赖关系总结一下。

---

## 10.1 复杂 KV 存储，是所有功能的基础

没有 array/hash/rbtree 的统一引擎层，就没有后面的：

* TTL 删除时不知道删谁
* 快照遍历时不知道怎么导出
* 复制同步时不知道如何恢复到哪套结构

所以第一基础是：

**统一 KV 引擎接口**

对应代码就在 `engine_set/get/mod/del/exist` 这一层。

---

## 10.2 key 定时删除依赖 KV 引擎抽象

TTL 模块自己不存 value，也不关心底层结构。
它只是记 `(engine, key, expire_at)`，到期后通过：

```c
engine_del(cur->engine, cur->key);
```

所以 TTL 依赖的是“统一删除接口”，而不是依赖某个具体结构。

---

## 10.3 全量/增量存储依赖统一命令模型

持久化没有自己定义二进制格式，而是直接复用 RESP 命令流：

* dump：把当前内存状态导出成一串 `SET/HSET/RSET + EXPIRE`
* aof：追加原始写命令
* recover：重新喂给 RESP 解析器执行

所以持久化依赖的是：

**命令即状态变更的统一表示**

---

## 10.4 主从复制依赖持久化能力

全量复制的核心不是“复制模块自己生成快照”，而是直接调用：

```c
kvs_snapshot_to_fp(fp)
```



这本来就是给 dump 用的。
也就是说：

* 全量同步依赖全量存储
* 增量同步依赖 AOF 一样的命令流模型

所以复制其实是建立在持久化基础上的再利用。

---

## 10.5 三种内存策略是横切模块

内存模块不属于业务链条上的某一步，它是整个系统的公共基础设施。
因为所有模块都用 `kvs_malloc/kvs_free`：

* array 分配 key/value
* hash 分配节点
* rbtree 分配节点
* expire 分配过期项
* reactor 分配发送节点

所以内存策略切换会影响整个系统，而不是某个局部功能。

---

## 10.6 批量处理依赖 RESP 流解析 + reactor 输入缓冲

批量处理并不是一套额外业务逻辑，而是：

* reactor 允许一次读取很多字节
* 输入缓冲保留未完整解析的数据
* `parse_resp_stream()` 循环解析多条命令

所以 batch/pipeline 是网络层和协议层配合出来的结果。

---

# 11. 按“程序设计过程”重述一遍这套实现路径

如果我是这个项目作者，这套系统大概就是这样一步步长出来的：

### 第一步，先实现一个能联网收命令的服务

* 用 epoll/reactor 接收连接
* 用 RESP 解析客户端命令
* 用 `handle_parsed_command()` 做统一调度

### 第二步，实现最基本 KV 能力

* 先做 array 版，验证功能链路跑通
* 再加 hash，得到更实用的高效存储
* 再加 rbtree，得到有序存储能力
* 通过 `engine_*` 做统一封装

### 第三步，补上 TTL

* 单独维护 expire 表，而不是侵入三种引擎
* 读路径上加 lazy expire
* 事件循环中加 active expire

### 第四步，补上持久化

* 写命令成功后写 AOF
* 定期或手动生成 dump
* 启动时 dump + aof 回放恢复

### 第五步，补上主从复制

* 从库起线程拉主库
* 主库收到 `REPLSYNC` 后发全量快照
* 写命令继续增量广播

### 第六步，统一内存策略

* 把所有 malloc/free 替换成 kvs_malloc/free
* 再做 libc/jemalloc/custom 三种后端切换
* 再暴露 MEMSTAT 观测效果

### 第七步，让协议层天然支持批量处理

* 让 `parse_resp_stream()` 支持多条命令连续解析
* 配合 reactor 输入缓冲，直接支持 pipeline

所以这 6 个功能不是平行堆上去的，而是一个非常自然的演化过程。

---

# 12. 这套实现里最值得你真正吃透的几个技术点

最后我挑几个最重要、最有“代码味”的点，帮你抓重点。

## 12.1 RESP 命令流被复用了三次

同一份 RESP 格式，既用于：

* 客户端请求
* AOF 持久化
* 主从增量复制 / 恢复回放

这是项目里最漂亮的统一设计之一。

## 12.2 TTL 没侵入三种引擎

TTL 表独立维护 `(engine, key)`，通过统一 `engine_del()` 删除。
这样三种存储结构都不用改节点定义。

## 12.3 全量复制和全量持久化共用一套快照逻辑

`kvs_snapshot_to_fp()` 一次实现，两边都能用。

## 12.4 内存模块是“全局横切能力”

通过 `kvs_malloc/free` 把 allocator 切换从“局部功能”提升成“全系统能力”。

## 12.5 批量处理不是单独做的，而是协议解析器天然具备

这是系统设计比“功能拼接”更高级的地方。
