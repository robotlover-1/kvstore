# KVStore 项目技术文档 - 详细实现分析

## 项目概述
KVStore 是一个高性能、多功能的键值存储系统，支持多种存储引擎、主从复制、TTL过期、内存策略切换和持久化功能。项目采用模块化设计，代码结构清晰，便于扩展和维护。

## 目录结构
```
include/kvstore/
├── kvstore.h          # 主要头文件，定义数据结构与函数原型
├── server.h           # 服务器相关定义
├── core/              # 核心网络处理模块
├── expire/            # 过期/TTL 模块
├── memory/            # 内存管理模块（三种后端）
├── persistence/       # 持久化模块
├── replication/       # 主从复制模块
├── storage/           # 存储引擎模块
└── utils/             # 工具函数

src/
├── main/kvstore.c     # 主程序入口，命令分发
├── core/              # 网络事件循环（reactor/proactor）
├── expire/            # 过期实现
├── memory/            # 内存后端实现
├── persistence/       # 持久化实现
├── replication/       # 主从复制实现
└── storage/           # 三种存储引擎实现
```

## 1. 复杂KV存储实现

### 1.1 三种存储引擎架构
项目实现了三种不同的存储引擎，用户通过命令前缀选择：

| 引擎类型 | 命令前缀 | 实现文件 | 适用场景 |
|---------|---------|---------|---------|
| 数组引擎 | 无前缀 | `kvs_array.c` | 简单场景，固定容量(1024) |
| 红黑树引擎 | R前缀 | `kvs_rbtree.c` | 需要有序遍历，O(log n)操作 |
| 哈希表引擎 | H前缀 | `kvs_hash.c` | 快速查找，O(1)平均时间复杂度 |

### 1.2 统一的引擎接口
通过 `engine_*()` 函数族提供统一的接口：
```c
static int engine_set(int engine, char *key, char *value);
static char *engine_get(int engine, char *key);
static int engine_mod(int engine, char *key, char *value);
static int engine_del(int engine, char *key);
static int engine_exist(int engine, char *key);
```

### 1.3 数组引擎实现细节
**数据结构**：
```c
typedef struct kvs_array_item_s {
    char *key;
    char *value;
} kvs_array_item_t;

typedef struct kvs_array_s {
    kvs_array_item_t *table;  // 固定大小数组
    int idx;                  // 当前索引
    int total;                // 总条目数
} kvs_array_t;
```
**特点**：
- 固定容量 `KVS_ARRAY_SIZE = 1024`
- 线性查找 `O(n)`，适合小规模数据
- 内存预分配，减少频繁分配

### 1.4 红黑树引擎实现细节
**数据结构**：
```c
typedef struct _rbtree_node {
    unsigned char color;
    struct _rbtree_node *right, *left, *parent;
    KEY_TYPE key;      // 支持char*或int
    void *value;
} rbtree_node;

typedef struct _rbtree {
    rbtree_node *root;
    rbtree_node *nil;  // 哨兵节点
} rbtree;
```
**特点**：
- 自平衡二叉搜索树
- 保证插入、删除、查找 O(log n)
- 支持有序遍历
- 实现标准红黑树操作：左旋、右旋、插入修复、删除修复

### 1.5 哈希表引擎实现细节
**数据结构**：
```c
typedef struct hashnode_s {
#if ENABLE_KEY_POINTER
    char *key;          // 动态分配
    char *value;        // 动态分配
#else
    char key[128];      // 固定大小
    char value[512];    // 固定大小
#endif
    struct hashnode_s *next;  // 链表解决冲突
} hashnode_t;

typedef struct hashtable_s {
    hashnode_t **nodes;  // 桶数组
    int max_slots;       // 哈希表大小
    int count;           // 元素数量
} hashtable_t;
```
**特点**：
- 链地址法解决哈希冲突
- 默认 `max_slots = 1024`
- 支持两种存储模式：动态分配或固定大小
- 哈希函数：`_hash(char *key, int size)`

### 1.6 引擎选择机制
在 `handle_parsed_command()` 中，通过命令前缀判断引擎：
```c
static int cmd_engine(const char *cmd) {
    if (cmd[0] == 'R') return KVS_ENGINE_RBTREE;
    if (cmd[0] == 'H') return KVS_ENGINE_HASH;
    return KVS_ENGINE_ARRAY;
}
```

## 2. 主从复制实现

### 2.1 架构设计
采用传统的master-slave异步复制模型：
- Master：接收写命令，广播给所有连接的slave
- Slave：主动连接master，同步全量和增量数据
- 支持读写分离：slave默认只读

### 2.2 Master端实现
**数据结构**：
```c
typedef struct conn_s {
    int fd;
    int is_replica;          // 标记是否为slave连接
    struct conn_s *next_replica;  // 链表连接所有slave
    // ... 其他字段
} conn_t;

conn_t *g_replicas = NULL;  // 所有slave连接链表
pthread_mutex_t g_repl_lock = PTHREAD_MUTEX_INITIALIZER;
```

**核心函数**：
1. `repl_add_slave()` - 添加slave连接
2. `repl_remove_slave()` - 移除slave连接
3. `repl_broadcast()` - 广播写命令给所有slave

**命令广播机制**：
```c
// 在 handle_parsed_command() 中
if (!from_replication && is_write_cmd(cmd)) {
    persist_append_raw(raw, rawlen);  // 本地持久化
    if (g_cfg.role == ROLE_MASTER) repl_broadcast(raw, rawlen);  // 广播
}
```

### 2.3 Slave端实现
**同步流程**：
1. 创建独立线程 `slave_thread()`
2. 循环尝试连接master
3. 发送 `REPLSYNC` 命令
4. 接收master的全量快照数据
5. 持续接收增量命令流

**复制线程** (`kvs_repl.c`)：
```c
static void *slave_thread(void *arg) {
    for (;;) {
        int fd = socket(...);
        connect(fd, master_address);
        send(fd, "REPLSYNC", ...);  // 请求同步
        
        // 接收全量快照
        // 持续接收增量命令
        parse_resp_stream(NULL, buf, &blen, 1);  // from_replication=1
    }
}
```

### 2.4 全量同步机制
当slave连接时，master执行 `queue_snapshot()`：
1. 发送 `FULLRESYNC` 简单字符串
2. 将当前数据快照到临时文件
3. 分块发送文件内容
4. 发送 `REPLDONE` 命令标记结束

### 2.5 读写控制
Slave只允许执行读命令：
```c
static int is_readonly_slave_blocked(const char *cmd) {
    return strcmp(cmd, "GET") && strcmp(cmd, "TTL") && strcmp(cmd, "EXIST") 
           && strcmp(cmd, "RGET") && ... ;  // 所有读命令
}
```

## 3. Key的定时删除实现

### 3.1 数据结构设计
**过期表结构**：
```c
#define KVS_EXPIRE_BUCKETS 1024

typedef struct kvs_expire_item_s {
    char *key;                  // 键名
    int engine;                 // 所属引擎
    long long expire_at_ms;     // 过期时间戳(毫秒)
    struct kvs_expire_item_s *next;  // 链表解决哈希冲突
} kvs_expire_item_t;

typedef struct kvs_expire_table_s {
    kvs_expire_item_t **buckets;  // 哈希桶数组
    int size;                    // 桶数量
} kvs_expire_table_t;
```

### 3.2 哈希函数
使用FNV-1a哈希算法：
```c
static unsigned int expire_hash(const char *key, int engine) {
    unsigned int h = 2166136261u;
    while (*key) {
        h ^= (unsigned char)(*key++);
        h *= 16777619u;
    }
    h ^= (unsigned int)engine;
    h *= 16777619u;
    return h % KVS_EXPIRE_BUCKETS;
}
```

### 3.3 过期操作API
1. **设置过期时间**：`kvs_expire_set()`
   - 计算过期时间戳：`now_ms + ttl_ms`
   - 插入或更新哈希表

2. **检查过期**：`kvs_expire_is_expired()`
   - 查找key对应的过期条目
   - 比较当前时间与过期时间戳

3. **获取剩余TTL**：`kvs_expire_ttl()`
   - 返回剩余秒数（向上取整）
   - 返回-1表示无过期设置
   - 返回-2表示已过期

4. **删除过期设置**：`kvs_expire_del()`/`kvs_expire_persist()`

### 3.4 被动过期检查
在每次操作前检查：
```c
static int try_expire(int engine, char *key) {
    if (kvs_expire_is_expired(&global_expire, engine, key)) {
        engine_del(engine, key);                  // 删除存储
        kvs_expire_del(&global_expire, engine, key); // 删除过期条目
        return 1;
    }
    return 0;
}
```

### 3.5 主动过期清理
后台周期性清理：
```c
int kvs_active_expire_cycle(int budget) {
    // budget: 本次清理的最大数量
    int removed = 0;
    long long now = kvs_now_ms();
    
    for (每个哈希桶 && removed < budget) {
        // 遍历链表，删除已过期的key
        // 同时从存储引擎中删除数据
    }
    return removed;  // 返回实际清理数量
}
```

**清理触发机制**：
在reactor主循环中每100ms执行一次：
```c
long long now = kvs_now_ms();
if (now - g_last_expire >= 100) {
    kvs_active_expire_cycle(32);  // 每次最多清理32个
    g_last_expire = now;
}
```

### 3.6 过期与持久化的集成
在数据快照时，会同时保存过期信息：
```c
static int maybe_emit_expire(FILE *fp, int engine, const char *key) {
    long long ttl = kvs_expire_ttl(&global_expire, engine, key);
    if (ttl < 0) return 0;  // 无过期设置
    char sec[32];
    snprintf(sec, sizeof(sec), "%lld", ttl);
    const char *cmd = engine == KVS_ENGINE_ARRAY ? "EXPIRE" 
                    : engine == KVS_ENGINE_RBTREE ? "REXPIRE" 
                    : "HEXPIRE";
    return emit_cmd3_fp(fp, cmd, key, sec);  // 写入持久化文件
}
```

## 4. 三种内存策略的实现与切换

### 4.1 内存后端架构
支持三种内存分配策略：
| 后端类型 | 标识 | 特点 |
|---------|------|------|
| libc | `KVS_MEM_LIBC` (0) | 标准glibc malloc/free |
| jemalloc | `KVS_MEM_JEMALLOC` (1) | 高性能内存分配器，需LD_PRELOAD加载 |
| custom | `KVS_MEM_CUSTOM` (2) | 自定义slab分配器+mmap大内存 |

### 4.2 配置与初始化
**启动参数**：
```bash
./kvstore --port 5000 --mem libc|jemalloc|custom
```

**初始化流程**：
1. 解析命令行参数，设置 `g_cfg.mem_backend`
2. 如果是jemalloc，执行 `kvs_mem_prepare_process()` 处理LD_PRELOAD
3. 调用 `kvs_mem_init()` 设置后端类型

### 4.3 Jemalloc特殊处理
由于jemalloc的TLS（线程局部存储）问题，需要特殊处理：
```c
static int ensure_jemalloc_process_image(const char *argv0, char **argv) {
    // 检查是否已激活
    if (getenv("KVS_MEM_JEMALLOC_ACTIVE")) return 0;
    
    // 查找jemalloc库路径
    const char *jemalloc_path = find_jemalloc_path();
    
    // 设置LD_PRELOAD环境变量
    setenv("LD_PRELOAD", jemalloc_path, 1);
    setenv("KVS_MEM_JEMALLOC_ACTIVE", "1", 1);
    
    // 重新execvp当前进程
    execvp(argv0, argv);
    return -1;
}
```

### 4.4 自定义内存分配器实现
**分层设计**：
1. **小内存分配** (≤1024B)：slab分配器
2. **大内存分配** (>1024B)：直接mmap
3. **回退机制**：使用系统malloc（当slab/mmap失败时）

**数据结构**：
```c
typedef struct small_chunk_s {
    uint32_t magic;           // 幻数：0xC0DEC0DE
    uint16_t class_idx;       // 所属size class索引
    uint16_t reserved;
    uint32_t request_size;    // 实际请求大小
    struct small_chunk_s *next;  // 空闲链表
} small_chunk_t;

typedef struct slab_page_s {
    void *mem;                // mmap分配的内存页
    size_t size;              // 页大小
    struct slab_page_s *next; // 链表
} slab_page_t;

typedef struct large_hdr_s {
    uint32_t magic;           // 幻数：0xC0DEC0DF
    uint32_t reserved;
    size_t request_size;      // 请求大小
    size_t mapping_size;      // 实际映射大小（页对齐）
} large_hdr_t;
```

**Size Classes**：
```c
static small_class_t classes[SMALL_CLASS_COUNT] = {
    {32, NULL, NULL, 0, 0, 0},   // class 0
    {64, NULL, NULL, 0, 0, 0},   // class 1
    {128, NULL, NULL, 0, 0, 0},  // class 2
    {256, NULL, NULL, 0, 0, 0},  // class 3
    {384, NULL, NULL, 0, 0, 0},  // class 4
    {512, NULL, NULL, 0, 0, 0},  // class 5
    {768, NULL, NULL, 0, 0, 0},  // class 6
    {1024, NULL, NULL, 0, 0, 0}, // class 7
};
```

**分配流程** (`custom_malloc()`)：
1. **小内存分配**：
   - 根据size找到对应的class索引
   - 从class的空闲链表中获取chunk
   - 如果空闲链表为空，调用 `slab_grow_locked()` 分配新页
   - 更新统计信息

2. **大内存分配**：
   - 计算总大小：`sizeof(large_hdr_t) + size`
   - 页对齐后通过mmap分配
   - 设置large header
   - 更新大内存统计

3. **回退分配**：
   - 当上述分配失败时，使用 `fallback_malloc()`
   - 基于系统malloc，添加fallback header
   - 更新回退统计

**释放流程** (`custom_free()`)：
1. 根据magic识别chunk类型
2. 小内存：放回对应class的空闲链表
3. 大内存：munmap释放映射
4. 回退内存：free释放

### 4.5 内存统计与监控
**MEMSTAT命令实现**：
```c
static int build_memstat_text(char *buf, size_t cap) {
    kvs_mem_stats_t st;
    if (kvs_mem_get_stats(&st) != 0) return -1;
    
    // 格式化输出所有统计信息
    // 包括：后端类型、分配调用次数、内存使用情况、碎片率等
}
```

**关键统计指标**：
1. **小内存统计**：
   - `current_small_inuse`：当前使用中的小内存块数
   - `peak_small_inuse`：峰值使用量
   - `total_small_page_bytes`：所有slab页总大小

2. **大内存统计**：

3. **性能指标**：
   - `internal_fragment_ppm`：内部碎片率（百万分之一）
   - `page_utilization_ppm`：页利用率（百万分之一）

通过函数指针机制提供统一接口：

## 5. 全量/增量存储实现

### 5.1 持久化架构
支持两种持久化方式：
1. **全量快照 (Snapshot/Dump)**：将内存数据完整保存到文件
2. **增量日志 (AOF)**：追加写入所有写命令

**文件配置**：
```c
typedef struct {
    char dump_path[256];  // 全量快照文件路径
    char aof_path[256];   // 增量日志文件路径
} kv_config_t;
```

### 5.2 全量快照实现
**快照流程** (`kvs_snapshot_to_fp()`)：
1. 遍历数组引擎的所有有效键值对
2. 遍历红黑树引擎的所有节点（中序遍历）
3. 遍历哈希表引擎的所有桶和链表节点
4. 为每个键值对生成对应的SET命令
5. 如果键有过期时间，生成对应的EXPIRE命令
6. 将所有命令写入文件

**命令序列化**：
```c
static int emit_cmd3_fp(FILE *fp, const char *cmd, const char *a1, const char *a2) {
    unsigned char buf[BUFFER_CAP];
    size_t n = resp_build_cmd3(buf, sizeof(buf), cmd, a1, a2);
    return fwrite(buf, 1, n, fp) == n ? 0 : -1;
}
```

**SAVE命令**：
```c
// 在handle_parsed_command()中
if (!strcmp(cmd, "SAVE")) {
    n = (persist_save_dump() == 0) 
        ? resp_simple_string(resp, sizeof(resp), "OK") 
        : resp_error(resp, sizeof(resp), "save failed");
}
```

### 5.3 增量日志实现
**AOF机制**：
1. **初始化**：`persist_init()` 打开AOF文件（追加模式）
2. **追加写入**：每次写命令执行后调用 `persist_append_raw()`
3. **同步刷盘**：`fflush(g_aof_fp)` 确保数据落盘

**写命令拦截**：
```c
// 在handle_parsed_command()中，写命令执行成功后
if (!from_replication && is_write_cmd(cmd)) {
    persist_append_raw(raw, rawlen);  // 记录到AOF
    if (g_cfg.role == ROLE_MASTER) repl_broadcast(raw, rawlen);
}
```

### 5.4 数据恢复
**恢复流程** (`persist_recover()`)：
1. **加载全量快照**：`replay_file(g_cfg.dump_path)`
2. **重放增量日志**：`replay_file(g_cfg.aof_path)`
3. **重放机制**：使用 `parse_resp_stream(NULL, buf, &len, 1)`，`from_replication=1`避免再次持久化

**文件重放实现**：
```c
static int replay_file(const char *path) {
    FILE *fp = fopen(path, "rb");
    unsigned char buf[BUFFER_CAP];
    size_t len = 0, n;
    
    while ((n = fread(buf + len, 1, sizeof(buf) - len, fp)) > 0) {
        len += n;
        parse_resp_stream(NULL, buf, &len, 1);  // 重放命令
    }
    fclose(fp);
    return 0;
}
```

### 5.5 持久化与复制的协同
1. **Master本地持久化**：所有写命令同时记录到AOF
2. **复制流持久化**：Slave接收的命令不写入本地AOF（避免循环）
3. **快照兼容性**：快照文件格式与复制流格式相同，可直接用于全量同步

## 6. 批量处理实现

### 6.1 网络层面的批量处理
**Reactory模式批量事件处理**：
```c
int n = epoll_wait(g_epfd, events, MAX_EVENTS, 100);
for (int i = 0; i < n; ++i) {
    // 批量处理就绪的事件
    // 每个连接可能有多个待处理命令
}
```

**输出缓冲队列**：
每个连接维护一个输出队列，支持批量发送：
```c
typedef struct out_node_s {
    unsigned char *data;  // 响应数据
    size_t len;           // 数据长度
    size_t sent;          // 已发送长度
    struct out_node_s *next;  // 下一个节点
} out_node_t;

typedef struct conn_s {
    out_node_t *out_head;  // 输出队列头
    out_node_t *out_tail;  // 输出队列尾
    // ...
} conn_t;
```

### 6.2 命令解析的批量处理
**RESP流式解析**：
`parse_resp_stream()` 函数能够处理缓冲区中的多个命令：
1. 识别RESP数组格式 (`*` 开头)
2. 批量解析多个bulk string
3. 批量分配内存存储参数
4. 一次性处理多个完整命令

**解析优化**：
- 支持inline命令（非RESP格式）的批量解析
- 缓冲区管理：已处理的数据被移除，未完整的数据保留

### 6.3 内存分配的批量优化
**Slab预分配**：
自定义内存分配器在 `slab_grow_locked()` 中批量分配多个chunk：
```c
int count = (int)(page_size / chunk_total);
if (count < 16) count = 16;
size_t alloc_size = chunk_total * (size_t)count;

void *mem = mmap(NULL, alloc_size, PROT_READ | PROT_WRITE, 
                 MAP_PRIVATE | MAP_ANONYMOUS, -1, 0);

// 初始化所有chunk并加入空闲链表
for (int i = 0; i < count; ++i) {
    small_chunk_t *chunk = (small_chunk_t *)(p + i * chunk_total);
    chunk->next = g_mem.classes[class_idx].free_list;
    g_mem.classes[class_idx].free_list = chunk;
}
```

### 6.4 过期清理的批量执行
**主动过期批量处理**：
`kvs_active_expire_cycle(int budget)` 支持批量清理：
- `budget` 参数控制单次最多清理的key数量
- 避免一次性清理过多key导致服务阻塞
- 分批遍历过期哈希表

### 6.5 复制流的批量处理
**Slave批量接收**：

for (;;) {
    ssize_t r = recv(fd, buf + blen, sizeof(buf) - blen, 0);
    if (r <= 0) break;
    blen += (size_t)r;

## 8. 性能优化设计

### 8.1 内存优化
1. **Slab分配器**：减少小内存分配碎片
2. **大内存mmap**：避免malloc开销，直接页对齐映射
3. **统计监控**：实时跟踪内存使用，优化分配策略

### 8.2 网络优化
1. **非阻塞IO**：所有socket设置为非阻塞模式
2. **输出缓冲队列**：避免多次系统调用
3. **事件驱动**：单线程处理大量并发连接

### 8.3 存储优化
1. **多引擎选择**：根据不同场景选择最优数据结构
2. **惰性过期删除**：被动检查避免额外开销
3. **批量持久化**：AOF追加写入，定期刷盘

### 8.4 复制优化
1. **异步复制**：不阻塞主线程操作
2. **增量同步**：全量后只同步写命令
3. **连接重试**：slave自动重连机制
